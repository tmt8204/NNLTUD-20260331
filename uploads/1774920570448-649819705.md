# Kiến trúc Microservices - SocialMedia Platform

> **Stack**: Spring Boot 3.x · SQL Server · MongoDB · Redis · RabbitMQ · MinIO/S3 · Docker · Kubernetes  
> **Cơ sở**: Phân tích từ monolith hiện tại (Spring Boot 4.0, MSSQL, local filesystem, raw WebSocket)

### 1.3 Dependency Matrix Giữa Các Module

```
Auth        ← [không phụ thuộc module khác]
User        ← Auth (JWT principal → userId)
Posts       ← User (lấy thông tin tác giả), Media (imageUrl)
Comments    ← Posts (postId FK), User (commenterId)
Reactions   ← Posts (postId FK), User (reactorId)
Friends     ← User (userId FK, friendId FK)
Chat        ← User (sender/receiver), Media (imageUrl, videoUrl)
Notifications ← User (actorId, targetId), Posts (postId ref), Friends (trigger)
Search      ← User (search by name), Posts (search by content)
Media       ← [không phụ thuộc - chỉ lưu file, trả URL]
Admin       ← User, Posts, Comments (cross-module reads/writes)
WebSocket   ← Auth (JWT handshake), Chat, Notifications, Friends
```

**Vấn đề chính trong monolith hiện tại**:
- `User` entity bị dùng bởi TẤT CẢ các module → tạo coupling toàn cục
- `Notification` được tạo trực tiếp trong `CommentService`, `ReactionService`, `FriendService` → tight coupling
- WebSocket sessions lưu trong `ConcurrentHashMap` → không scale ngang được
- Upload files lên filesystem local → không HA, không CDN
- `content` field trong Post giới hạn 100 ký tự → cần mở rộng

## 3. THIẾT KẾ GIAO TIẾP GIỮA CÁC SERVICE

### 3.1 Sync Communication (REST / gRPC)

```
┌─────────────────────────────────────────────────────────┐
│ Synchronous REST Calls (cần response ngay)              │
├──────────────────────────┬──────────────────────────────┤
│ Caller                   │ Called Service               │
├──────────────────────────┼──────────────────────────────┤
│ social-service           │ user-service                 │
│   → GET /internal/users/{id}  → lấy displayName, avatar │
│                                                         │
│ admin-service            │ user-service, social-service  │
│   → GET /internal/users/{id}  → thông tin user để ban   │
│   → DELETE /internal/posts/{id} → xóa bài đăng          │
│                                                         │
│ api-gateway              │ auth-service                 │
│   → POST /internal/auth/validate-token  → verify JWT    │
│                                                         │
│ chat-service             │ user-service                 │
│   → GET /internal/users/{id}  → tên, avatar sender      │
├──────────────────────────┴──────────────────────────────┤
│ LƯU Ý: Gọi /internal/** được đặt trong network nội bộ  │
│ (không expose qua API Gateway), dùng service-to-service │
│ API key hoặc mTLS để bảo mật                            │
└─────────────────────────────────────────────────────────┘
```

**Resilience pattern** cho sync calls:
- **Circuit Breaker** (Resilience4j): nếu user-service down, social-service không crash
- **Retry** với exponential backoff (tối đa 3 lần)
- **Fallback**: trả về cached data hoặc placeholder user info
- **Timeout**: tối đa 3 giây mỗi internal call

### 3.2 Async Communication (RabbitMQ Events)

```
EVENT FLOW - RabbitMQ Exchanges & Queues

Exchange: post.events (topic)
├── post.created      → notification-service (thông báo followers)
│                     → search-service (index bài mới)
├── post.deleted      → search-service (remove index)
│                     → media-service (cleanup media files)
└── post.commented    → notification-service (thông báo tác giả)
    post.reacted      → notification-service (thông báo tác giả)

Exchange: friend.events (topic)
├── friend.requested  → notification-service (thông báo người nhận)
│                     → ws-gateway (push real-time)
├── friend.accepted   → notification-service
│                     → ws-gateway (push real-time)
└── friend.removed    → social-service (cleanup feed access)

Exchange: media.tasks (direct)
├── media.uploaded    → media-service (generate thumbnails)
└── media.deleted     → media-service (delete from S3)

Exchange: admin.events (topic)
├── user.banned       → auth-service (invalidate tokens)
│                     → ws-gateway (disconnect sessions)
└── post.moderated    → social-service (mark deleted)
    
Exchange: chat.events (topic)
└── message.sent      → notification-service (push nếu offline)
                      → ws-gateway (broadcast real-time)
```

**Event Message Format (JSON)**:
```json
{
  "eventId": "uuid-v4",
  "eventType": "post.created",
  "timestamp": "2026-03-23T10:00:00Z",
  "sourceService": "social-service",
  "payload": {
    "postId": 123,
    "authorId": 456,
    "content": "Hello world!"
  }
}
```

### 3.3 Authentication & Authorization Flow

```
1. CLIENT → API Gateway:
   POST /auth/login   → forward đến auth-service
   ← JWT {userId, email, role, exp}

2. CLIENT → API Gateway (any secured endpoint):
   Header: Authorization: Bearer <jwt>
   
3. API Gateway:
   a. Extract JWT header
   b. Call auth-service: POST /internal/auth/validate
      hoặc verify locally nếu giữ public key (RS256 recommended)
   c. Inject headers: X-User-Id, X-User-Role, X-User-Email
   d. Forward request to downstream service

4. Downstream service:
   - Đọc X-User-Id từ header (không decode JWT nữa)
   - Áp dụng business authorization (ví dụ: chỉ owner mới sửa post)

LƯU Ý bảo mật:
- Dùng RS256 thay vì HS256: auth-service giữ private key, 
  tất cả service khác chỉ cần public key để verify
- JWT nên có short expiry (15-30 min) + refresh token
- Refresh token lưu trong Redis với TTL
```

### 4.1 Database Per Service

#### auth-service → SQL Server: `auth_db`
```sql
-- Table: users (credentials only, không có profile data)
CREATE TABLE users (
    user_id       INT IDENTITY PRIMARY KEY,
    email         VARCHAR(120) UNIQUE NOT NULL,
    user_name     VARCHAR(50)  UNIQUE NOT NULL,
    pass_word     VARCHAR(255) NOT NULL,       -- BCrypt
    status        BIT DEFAULT 1,               -- 0=banned
    role_id       INT NOT NULL,
    created_at    DATETIME2 DEFAULT GETDATE()
);

-- Table: roles
CREATE TABLE roles (
    role_id   INT IDENTITY PRIMARY KEY,
    role_name VARCHAR(50) UNIQUE NOT NULL      -- USER, ADMIN
);

-- Table: refresh_tokens (thay cho in-memory)
CREATE TABLE refresh_tokens (
    token_id    UNIQUEIDENTIFIER PRIMARY KEY DEFAULT NEWID(),
    user_id     INT NOT NULL,
    token_hash  VARCHAR(255) NOT NULL,
    expires_at  DATETIME2 NOT NULL,
    revoked     BIT DEFAULT 0
);
```

### 4.2 Data Consistency Patterns

```
Thách thức: Mỗi service có DB riêng → không dùng JOIN hoặc FK cross-service được

Giải pháp:

1. SAGA Pattern (cho long-running transactions):
   Ví dụ: User đăng ký → Auth tạo user → pub event →
   User service nhận event → tạo profile record

2. Event Sourcing (cho audit trail):
   Admin ban user:
   - admin-service publish event: UserBannedEvent
   - auth-service subscribe → flag user banned → invalidate tokens
   - ws-gateway subscribe → force disconnect user

3. CQRS (tách Read/Write):
   social-service → write posts
   search-service → read index (Elasticsearch, sync qua events)

4. API Composition (thay JOIN):
   Khi hiển thị Feed:
   - social-service trả list {postId, userId, content, ...}
   - api-gateway / BFF enrichment: gọi user-service lấy {userName, avatar}
   - Merge và trả client
```

### 4.3 Media & File Storage

```
Storage Architecture:

Client → api-gateway → media-service → MinIO/AWS S3
                                     ↓
                               Async event: media.tasks
                                     ↓
                          media-service worker:
                          - Resize ảnh (thumbnail 200x200, medium 800x600)
                          - Extract video thumbnail
                          - Virus scan (ClamAV)
                          - Update CDN URL trong DB

CDN Layer:
- MinIO self-hosted (phát triển/staging)
- AWS S3 + CloudFront (production)
- URL pattern: https://cdn.socialmedia.com/avatars/{uuid}.webp

Upload Flow:
1. Client request presigned URL: POST /api/media/presign
2. media-service trả: { uploadUrl, fileId }
3. Client upload trực tiếp lên S3 (giảm tải server)
4. media-service nhận S3 event notification → xử lý async
5. media-service update database với CDN URL
6. media-service publish event: media.ready { fileId, cdnUrl }
7. Calling service (social/user/chat) nhận event, update record
```

## 9. MIGRATION GUIDE - TỪ MONOLITH SANG MICROSERVICE

### 9.1 Chiến Lược Migration: Strangler Fig Pattern

```
Giai đoạn 1: Song Song vận hành (Strangler Fig)
─────────────────────────────────────────────────
Monolith vẫn chạy, tách từng service một:

Monolith :8080
    ↑
API Gateway :80
    │
    ├── /api/auth/** → auth-service :8081   ← TÁCH TRƯỚC
    ├── /api/users/** → user-service :8082  ← TÁCH SAU
    └── /** → Monolith :8080               ← Phần còn lại

Giai đoạn 2: Tách dần từng service
Giai đoạn 3: Tắt monolith khi đã chuyển hết

Ưu tiên tách theo thứ tự:
1. auth-service (ít dependency nhất)
2. media-service (stateless nhất)
3. user-service
4. notification-service
5. social-service (nhiều dependency nhất)
6. chat-service
7. admin-service
```

### 9.3 Data Migration Scripts

#### SQL Server → MongoDB Migration (Chat)
```javascript
// migrate-messages.js (Node.js migration script)
const sql = require('mssql');
const { MongoClient } = require('mongodb');

async function migrateMessages() {
    const sqlConn = await sql.connect({
        server: 'localhost',
        database: 'SocialMedia',
        user: 'sm_user',
        password: 'StrongPass@123',
        options: { trustServerCertificate: true }
    });
    
    const mongoClient = new MongoClient('mongodb://admin:MongoPass@123@localhost:27017');
    await mongoClient.connect();
    const db = mongoClient.db('chat_db');
    
    // Migrate messages in batches
    const BATCH_SIZE = 1000;
    let offset = 0;
    
    while (true) {
        const result = await sqlConn.query(`
            SELECT TOP ${BATCH_SIZE} message_id, sender_id, received_id, 
                   group_id, content, image_url, video_url, sent_at, read_at, status
            FROM message
            ORDER BY message_id
            OFFSET ${offset} ROWS
        `);
        
        if (result.recordset.length === 0) break;
        
        const docs = result.recordset.map(row => ({
            legacyId: row.message_id,
            conversationId: row.group_id 
                ? `group_${row.group_id}` 
                : [row.sender_id, row.received_id].sort().join('_'),
            senderId: row.sender_id,
            receiverId: row.received_id,
            groupId: row.group_id ? String(row.group_id) : null,
            content: row.content,
            mediaUrl: row.image_url || row.video_url || null,
            mediaType: row.video_url ? 'VIDEO' : (row.image_url ? 'IMAGE' : null),
            status: row.status ? 'READ' : 'SENT',
            sentAt: row.sent_at,
            readAt: row.read_at
        }));
        
        await db.collection('messages').insertMany(docs, { ordered: false });
        offset += BATCH_SIZE;
        console.log(`Migrated ${offset} messages...`);
    }
    
    await sqlConn.close();
    await mongoClient.close();
    console.log('Migration complete!');
}

migrateMessages().catch(console.error);
```

## 11. BẢO MẬT

### 11.1 Checklist Bảo Mật

```
JWT Security:
✓ Dùng RS256 (asymmetric) thay HS256
✓ Short expiry: Access token 15 phút, Refresh token 7 ngày
✓ Refresh token rotation: mỗi lần dùng → cấp token mới + revoke cái cũ
✓ Lưu refresh token hash (bcrypt) không phải raw token

API Gateway Security:
✓ Rate limiting: 100 req/min per IP, 1000 req/min per user
✓ JWT validation tại gateway (không để service tự validate)
✓ Inject X-User-Id header sau khi validate (trusted header)
✓ Internal endpoints (/internal/**) chỉ accessible trong cluster network
✓ mTLS cho service-to-service communication

Input Validation:
✓ Validate tất cả input tại controller (Jakarta Bean Validation)
✓ Sanitize HTML content (chống XSS) trong post/comment content
✓ File type validation tại media-service (magic bytes, không chỉ extension)
✓ Max file size enforcement

Database Security:
✓ Mỗi service có DB user riêng với quyền tối thiểu
✓ Secrets quản lý qua Vault / Kubernetes secrets (không hardcode)
✓ SQL queries qua JPA/Hibernate (chống SQL injection)
✓ Encrypt sensitive fields (email, phone) nếu cần GDPR compliance
```
